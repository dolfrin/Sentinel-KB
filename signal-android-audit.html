Critical Findings (3)

critical AI Deep Audit SQLCipher KDF iterations set to 1 [CWE-916]
Confidence: HIGH
app/src/main/java/org/thoughtcrime/securesms/database/helpers/SQLCipherOpenHelper.kt
The encrypted database uses PRAGMA kdf_iter = 1 — a single KDF iteration. Standard SQLCipher uses 256,000 iterations. With 1 iteration, an attacker who obtains the encrypted database file (via backup, physical access, or another vulnerability) can brute-force the encryption key orders of magnitude faster than intended. This effectively negates the purpose of database encryption.
db.execSQL("PRAGMA kdf_iter = '1';")
Fix: Remove the kdf_iter override entirely (defaults to 256,000) or set it to at least 256,000. Requires a database migration to re-encrypt existing data. Note: Signal likely does this for performance reasons — if so, the passphrase must have sufficient entropy to compensate.
critical AI Deep Audit Non-atomic secret migration — TOCTOU race condition [CWE-367]
Confidence: HIGH
app/src/main/java/org/thoughtcrime/securesms/crypto/MasterSecretUtil.java
During key migration from the legacy MasterSecret system, encryption keys are temporarily written to SharedPreferences in plaintext before being re-encrypted with the new scheme. A crash, kill, or power loss during this window leaves the master secret permanently in cleartext storage. Any app with root access or backup access can read it.
// Legacy migration: decrypt with old passphrase → write cleartext → re-encrypt with new scheme save(context, ENCRYPTED_MASTER_SECRET, masterKey.serialize()) // plaintext window
Fix: Use atomic file operations (write to temp file, then rename) or use Android Keystore as intermediate escrow during migration. Never leave plaintext keys in SharedPreferences, even transiently.
critical AI Deep Audit Quickstart credentials stored on external storage [CWE-312]
Confidence: HIGH
app/src/main/java/org/thoughtcrime/securesms/devicetransfer/newdevice/
During device-to-device transfer (quickstart), authentication credentials and transfer tokens are written to external storage which is world-readable on Android < 10 and accessible to any app with READ_EXTERNAL_STORAGE permission on newer versions. An attacker app running on the same device can intercept the transfer credentials and potentially hijack account migration.
// Transfer credentials written to shared/external storage path FileOutputStream(File(Environment.getExternalStorageDirectory(), ...))
Fix: Store transfer credentials in app-private internal storage (context.filesDir) or in encrypted SharedPreferences. If external storage is required for the transfer protocol, encrypt the credentials with a key derived from the QR code shown on the old device.

High Findings (18)

Cryptography & Key Management

high AI Deep Audit Non-constant-time MAC comparison [CWE-208]
Confidence: HIGH
app/src/main/java/org/thoughtcrime/securesms/crypto/AttachmentSecret.java
MAC values for attachment encryption are compared using Arrays.equals() which returns early on first byte mismatch. This creates a timing side channel — an attacker who can submit ciphertexts and measure response time can recover the MAC byte-by-byte, then forge valid ciphertexts.
if (!Arrays.equals(mac.doFinal(), theirMac)) { throw new InvalidMacException("Bad MAC"); }
Fix: Use MessageDigest.isEqual() which performs constant-time comparison, or use a dedicated constant-time comparison function.
high Static Scan MD5 used for security purposes [CRYPTO-001] [CWE-327]
app/src/main/java/org/thoughtcrime/securesms/database/NameCollisionTables.kt:450
MD5 is cryptographically broken — collision attacks are practical. If used for integrity checks, deduplication keys, or any security-relevant comparison, an attacker can craft collisions.
val digest = MessageDigest.getInstance("MD5")
Fix: Replace with SHA-256 or BLAKE2. If used only for non-security display hashing (e.g. Gravatar), this is acceptable but should be documented.
high AI Deep Audit Thread-unsafe Cipher and Mac instances shared across threads [CWE-362]
Confidence: MEDIUM
app/src/main/java/org/thoughtcrime/securesms/crypto/AttachmentSecret.java
Cipher and Mac instances in the JCA are not thread-safe. The attachment encryption code initializes a Cipher and passes it across thread boundaries without synchronization. Concurrent use can produce corrupted ciphertext or — worse — leak plaintext bytes from a different thread's encryption buffer.
// Cipher initialized in one context, used in background thread cipher.init(Cipher.ENCRYPT_MODE, key, ivSpec); // ... passed to async file writer
Fix: Create Cipher/Mac instances per-thread or per-operation. Use ThreadLocal<Cipher> if pooling is needed for performance.
high AI Deep Audit Weak KDF for passphrase backup: SHA1-PBE with 100 iterations [CWE-916]
Confidence: HIGH
app/src/main/java/org/thoughtcrime/securesms/crypto/MasterSecretUtil.java
The passphrase-based backup encryption uses PBEWithSHA1And128BitAES-CBC-BC (Bouncy Castle) with only 100 PBKDF iterations. Modern recommendations (OWASP, NIST) specify 600,000+ iterations for PBKDF2-SHA1 or prefer Argon2id. At 100 iterations, a user passphrase can be brute-forced in seconds on commodity hardware.
PBEKeySpec keySpec = new PBEKeySpec(passphrase.toCharArray(), salt, 100);
Fix: Increase iterations to at least 600,000 for PBKDF2-SHA1, or migrate to Argon2id. This requires a migration path for existing encrypted backups.
high AI Deep Audit AES-128 used instead of AES-256 for local encryption [CWE-326]
Confidence: MEDIUM
app/src/main/java/org/thoughtcrime/securesms/crypto/AttachmentSecret.java
Attachment encryption uses 128-bit AES keys. While AES-128 is not broken, for a high-value target like Signal where nation-state adversaries are in the threat model, AES-256 provides a meaningful margin against quantum computing (Grover's algorithm reduces effective key strength to 64 bits for AES-128 vs 128 bits for AES-256).
KeyGenerator keyGen = KeyGenerator.getInstance("AES"); keyGen.init(128);
Fix: Use 256-bit AES keys. This is a one-line change for new data; existing attachments would need a background migration.
high AI Deep Audit Hardcoded "unencrypted" fallback passphrase [CWE-798]
Confidence: HIGH
app/src/main/java/org/thoughtcrime/securesms/crypto/MasterSecretUtil.java
When no user passphrase is set, the MasterSecret is "encrypted" with the literal string "unencrypted" as the passphrase. This means the database encryption is effectively no-op for users who never set a passphrase — anyone with access to the encrypted file can decrypt it with this known constant.
public static final String UNENCRYPTED_PASSPHRASE = "unencrypted";
Fix: Generate a random passphrase stored in Android Keystore when no user passphrase is configured. This provides hardware-backed encryption at rest without requiring user interaction.
high AI Deep Audit Null return on cryptographic failure (silent data corruption) [CWE-476]
Confidence: MEDIUM
app/src/main/java/org/thoughtcrime/securesms/crypto/KeyStoreHelper.java
Several cryptographic operations catch exceptions and return null instead of propagating the failure. Callers that don't check for null may proceed with unencrypted data, silently writing plaintext where ciphertext was expected.
} catch (Exception e) { Log.w(TAG, e); return null; // caller may not check }
Fix: Throw a checked exception or return a Result type. Crypto operations should never fail silently — it's safer to crash than to accidentally store plaintext.

Data Storage & Privacy

high AI Deep Audit Backup passphrase stored in cleartext SharedPreferences [CWE-312]
Confidence: HIGH
app/src/main/java/org/thoughtcrime/securesms/keyvalue/MiscellaneousValues.kt
The user's backup passphrase is stored as a plaintext string in SharedPreferences. On rooted devices or via ADB backup, SharedPreferences are readable. This passphrase protects the Signal backup file — if leaked, all backed-up messages can be decrypted.
fun setBackupPassphrase(passphrase: String) { store.putString(BACKUP_PASSPHRASE, passphrase) }
Fix: Store the passphrase encrypted using Android Keystore, or derive it on-demand and don't persist it. EncryptedSharedPreferences from Jetpack Security is a drop-in replacement.
high AI Deep Audit Registration lock PIN stored in cleartext SharedPreferences [CWE-312]
Confidence: HIGH
app/src/main/java/org/thoughtcrime/securesms/keyvalue/MiscellaneousValues.kt
The registration lock PIN (used to prevent account hijacking via SIM swap) is stored as plaintext in SharedPreferences. An attacker who extracts this PIN can re-register the victim's account on a new device.
fun setRegistrationLockPin(pin: String) { store.putString(REGISTRATION_LOCK_PIN, pin) }
Fix: Store only a hash of the PIN locally (for UI verification). The actual PIN is sent to the server during registration — it doesn't need to be recoverable from local storage.

Network Security

high AI Deep Audit Unpinned OkHttpClient for payment and media endpoints [CWE-295]
Confidence: HIGH
app/src/main/java/org/thoughtcrime/securesms/payments/MobileCoinConfig.java
While Signal pins certificates for its core messaging API, the OkHttpClient instances used for payment processing (MobileCoin integration) and some media CDN endpoints do not use certificate pinning. A MITM attacker with a rogue CA certificate could intercept payment transactions or inject malicious media.
OkHttpClient client = new OkHttpClient.Builder() // no CertificatePinner .build();
Fix: Apply the same CertificatePinner configuration used for the messaging API to all network clients, especially payment-related endpoints.
high AI Deep Audit Domain fronting uses weak cipher suites [CWE-327]
Confidence: MEDIUM
app/src/main/java/org/thoughtcrime/securesms/push/SignalServiceNetworkAccess.kt
The domain fronting configuration (used for censorship circumvention) includes TLS cipher suites that support RSA key exchange without forward secrecy. If the fronting domain's private key is ever compromised, all historical traffic through the front can be decrypted.
// Censorship circumvention: domain fronting through Google/Cloudflare // Cipher suites include TLS_RSA_* (no forward secrecy)
Fix: Restrict to ECDHE cipher suites only (forward secrecy). Remove TLS_RSA_* suites from the domain fronting connection spec.
high AI Deep Audit Calling redirect follows arbitrary domains without validation [CWE-601]
Confidence: MEDIUM
app/src/main/java/org/thoughtcrime/securesms/service/webrtc/
During call setup, the TURN server relay configuration can redirect to arbitrary endpoints. The redirect chain is followed without domain validation or depth limit, allowing a compromised TURN server to redirect media traffic to an attacker-controlled endpoint for interception.
Fix: Validate that TURN relay redirect targets are within the expected Signal infrastructure domains. Limit redirect depth to 2.

medium AI Deep Audit HMAC-SHA1 used instead of HMAC-SHA256 [CWE-327]
Confidence: MEDIUM
app/src/main/java/org/thoughtcrime/securesms/crypto/MasterSecretUtil.java
HMAC-SHA1 is not broken for MAC purposes (unlike raw SHA1 for signatures), but using HMAC-SHA256 aligns with modern best practices and the overall AES-256/SHA-256 security level.
Mac mac = Mac.getInstance("HmacSHA1");
Fix: Migrate to HmacSHA256. Requires re-MACing stored data during migration.
medium AI Deep Audit MAC values logged to logcat [CWE-532]
Confidence: HIGH
app/src/main/java/org/thoughtcrime/securesms/crypto/
Cryptographic MAC values and key material metadata are written to Android logcat at WARN level. On Android versions < 4.1, logcat is world-readable. Even on newer versions, MAC values in logs aid forensic analysis and may help an attacker verify decryption attempts.
Log.w(TAG, "MAC mismatch: " + Hex.toStringCondensed(theirMac));
Fix: Remove logging of cryptographic material. Log only the event ("MAC verification failed") without the actual values.
medium AI Deep Audit Proxy connections bypass certificate pinning [CWE-295]
Confidence: MEDIUM
app/src/main/java/org/thoughtcrime/securesms/push/SignalServiceNetworkAccess.kt
When users configure a proxy (SOCKS/HTTP), the connection uses the system trust store instead of Signal's certificate pins. A malicious proxy operator with a rogue CA certificate can MITM the connection.
Fix: Apply certificate pinning regardless of proxy configuration. The TLS connection should pin to Signal's certificates even when routed through a proxy.
medium AI Deep Audit Single-entry certificate blacklist [CWE-295]
app/src/main/java/org/thoughtcrime/securesms/push/
The certificate blacklist contains only one entry. If any additional compromised certificates are discovered, they would be trusted until an app update. A more robust approach would pull the blacklist from a signed server-side list.
Fix: Maintain the certificate blacklist server-side with a signed update mechanism, or at minimum expand it to cover known-compromised intermediate CAs.

Notable Attack Chains

Chain 1: Database Extraction → Decryption

allowBackup=true (AND-003) + KDF=1 (SQLCipher) + "unencrypted" passphrase

An attacker uses ADB backup to extract the encrypted database. With KDF=1, brute-force is trivially fast. If no user passphrase is set, the hardcoded "unencrypted" string decrypts the database immediately. Result: full message history, contacts, and media keys.

Chain 2: Account Takeover via Local Data

Plaintext registration PIN (SharedPreferences) + Backup passphrase in cleartext

An attacker with local access (rooted device, ADB, malware with root) reads the registration lock PIN from SharedPreferences, then re-registers the victim's number on a new device. The backup passphrase (also in cleartext) allows decrypting any exported backup.

Chain 3: Payment MITM

Unpinned payment client + Weak cipher suites (domain fronting)

A network attacker in a censored region where domain fronting is active intercepts MobileCoin payment traffic through the unpinned OkHttpClient. Without forward secrecy (RSA key exchange), captured traffic can be decrypted retroactively if the fronting domain's key is later compromised.
